# 2026-04-03 更新请求：自动生成 Agent 部署命令与实际部署入口脱节

## 文档目的

这份文档只描述问题、原因、影响范围、建议改法和验收口径。

这不是一份“立即修改线上”的执行记录，也不是对现有部署的回滚建议。

目标是让后续接手的人，能够基于这份文档直接理解为什么会出现“后台自动生成的 Agent 部署命令，和真实可用入口不一致”的问题，以及后续应该怎么改。

## 本次实际场景复盘

2026-04-03 的实际部署场景如下：

- 原机器上已经有旧项目占用了 9191 和 9192。
- 为了与旧项目并存，NetworkStatus-Rabbit-NG 实际部署到了 9193 和 9194。
- Web 入口没有直接暴露 9193，而是通过 oneinstack 反代到了 https://network-status.wingsrabbit.com 的 443。
- Agent 实际连接的中心端口不是 9192，而是 9194。
- 结果是：后台“节点管理”里自动生成的 Agent 部署命令，仍然指向默认的 9191 和 9192，不能直接使用。

最终实际可用的部署方式，是人工按真实入口去改命令，而不是直接使用后台自动生成结果。

这说明当前系统的“自动生成部署命令”逻辑，没有跟随真实部署入口变化而变化。

## 问题现象

从使用者视角看，问题是这样的：

- 前端访问入口已经变成了 443 域名。
- Docker 外部映射端口也已经改成了 9193 和 9194。
- 但后台生成的 Agent 部署命令，仍然默认写成 9191 和 9192。
- 用户会误以为“前端已经改了，后台应该自动知道现在的部署入口”。
- 实际上并不会，因为当前自动生成命令的逻辑不在前端，而在后端，而且是写死的。

更进一步地说，这个问题不是单纯“前端显示错了”，而是“生成逻辑本身和实际部署拓扑脱节”。

## 为什么前端改了，后台命令不会跟着变

核心原因很简单：

- 前端并不自己拼接部署命令。
- 前端只是调用后端接口，拿到后端已经生成好的字符串，再展示出来。

当前前端接口定义位于：

- web/src/api/nodes.ts

其中 `getDeployCommand(id)` 只是请求：

- `/nodes/{id}/deploy-command`

也就是说：

- 命令内容由后端决定。
- 前端只负责展示。
- 反代到 443、域名变化、入口变化，这些如果没有显式进入后端配置，后端是不会自动知道的。

所以“前端入口改了”这件事，本身不会改变后台返回的部署命令。

## 当前问题的直接根因

### 1. 后端部署命令生成逻辑写死了默认端口

当前关键代码位于：

- server/api/nodes.py

现有实现核心逻辑如下：

```python
host = request.host.split(':')[0]
agent_port = 9192

command = (
    f'bash <(curl -sL http://{host}:9191/api/install-agent.sh) agent '
    f'--server {host} --port {agent_port} --node-id {node.id} --token {token}'
)
docker_command = (
    f'docker run -d --restart=always --name ns-agent --net=host '
    f'nsr-agent '
    f'--server {host} --port {agent_port} --node-id {node.id} --token {token}'
)
docker_command_with_listen = (
    f'docker run -d --restart=always --name ns-agent --net=host '
    f'nsr-agent '
    f'--server {host} --port {agent_port} --node-id {node.id} --token {token} '
    f'--listen-port 9192'
)
```

这里至少有四个硬编码问题：

- 安装脚本下载地址写死为 9191。
- Agent 连接端口写死为 9192。
- `request.host.split(':')[0]` 会主动把端口去掉。
- `docker_command_with_listen` 的 `--listen-port` 也写死为 9192。

这意味着只要不是“标准默认部署”，生成命令就很容易失真。

### 2. Docker Compose 明明支持改端口，但命令生成逻辑不知道

当前 Docker Compose 位于：

- docker-compose.yml

这里已经支持通过环境变量改外部映射端口：

```yaml
ports:
  - "${NSR_WEB_PORT:-9191}:80"

ports:
  - "${NSR_AGENT_PORT:-9192}:5000"
```

也就是说：

- 系统的实际宿主机端口，本来就是允许变化的。
- 但 `deploy-command` 接口没有读取这些有效配置。
- 于是“运行时端口”和“生成命令时端口”变成了两个互不关联的世界。

### 3. 反向代理场景下，浏览器入口和 Agent 入口本来就可能不是同一个地址

这是本次问题最关键的业务现实。

在实际部署中，通常至少有两类入口：

- 浏览器访问入口。
- Agent 连接中心的入口。

这两个入口可能完全不同。

例如本次真实场景：

- 浏览器访问入口是 https://network-status.wingsrabbit.com。
- Agent 实际连接入口是 43.133.15.207:9194。

这意味着系统不能再假设：

- 浏览器访问地址 = Agent 连接地址。
- Web 端口 = Agent 端口。

而当前后端生成逻辑，实际上仍然在默认这种假设成立。

### 4. `request.host.split(':')[0]` 会把非默认端口信息直接丢掉

即使不考虑反代，只考虑“同一个 IP 但换了端口”的情况，这段逻辑也有问题。

例如：

- 用户访问的是 `http://1.2.3.4:9193`。
- `request.host` 本来有端口信息。
- 但 `split(':')[0]` 后，端口被直接丢掉。

接下来代码再手动写死 `:9191` 和 `:9192`，就彻底和真实入口脱离了。

### 5. `docker_command_with_listen` 把本地监听端口和中心连接端口混为一谈

当前代码把：

- Agent 连接中心的端口。
- Agent 本地 `--listen-port`。

都默认绑定到 9192 语义附近。

但这两个端口实际上不是同一个概念：

- `--port` 是 Agent 要连去中心服务器的端口。
- `--listen-port` 是 Agent 本机为了内部探测而开启的本地监听端口。

二者不应该被简单硬编码成同一个默认值，更不应该在有端口冲突或已有业务时被强行写死。

## 相关代码和文档中还存在的同类默认假设

这个问题不是只有一个点。

### 1. 一键安装脚本 `install.sh` 仍然按 9191 / 9192 思维输出信息

文件：

- install.sh

当前脚本中仍然写有以下默认逻辑：

- Agent 安装示例默认 `--port 9192`。
- 服务端就绪探测使用 `http://localhost:9191/api/auth/login`。
- 安装完成提示输出 `http://IP:9191`。
- 安装完成提示输出 Agent 端口 `IP:9192`。

这会导致两个问题：

- 即使实际服务是用自定义端口启动，脚本输出和健康检查信息也可能不准确。
- 新用户会继续被文档和脚本提示带回默认端口思维。

### 2. Agent 安装脚本 `scripts/install-agent.sh` 自身也有端口语义耦合

文件：

- scripts/install-agent.sh

这个脚本的关键问题是：

- 它只收一个 `--port`。
- 这个 `--port` 同时被用于“下载 agent package”和“运行时连接中心”。

脚本内逻辑如下：

```bash
AGENT_URL="http://${SERVER}:${PORT}/api/agent-package.tar.gz"
```

以及启动时：

```bash
exec python3 -m agent.main \
    --server "${SERVER}" \
    --port "${PORT}" \
```

这说明当前脚本默认认为：

- Agent 包下载入口。
- Agent 运行时连接入口。

是同一个 host:port。

这在“Web 面板走 443 反代，Agent 通道走单独 9194”这种拓扑下，天然就不是普适假设。

### 3. Agent 当前只能表达 `http://host:port`，不能表达完整 URL 或 scheme

文件：

- agent/config.py

当前实现：

```python
@property
def server_url(self):
    return f'http://{self.server}:{self.port}'
```

这意味着当前 Agent 命令行只能表达：

- 一个 host。
- 一个 port。
- 且 scheme 被写死为 `http`。

这带来的限制是：

- 当前系统更适合“Agent 连一个明确的 HTTP 端口”。
- 如果以后想让 Agent 也走 443、HTTPS、甚至 WSS，需要扩展为支持 scheme 或完整 URL。

也就是说，后续若要把“Agent 入口也统一到 443 域名”作为正式目标，不能只改前端或命令模板，还要改 Agent 配置表达能力。

## 问题的本质

本质上，这不是一个简单的“端口写错了”的 bug。

这是一个“系统缺少公共入口抽象”的设计问题。

当前代码把以下几层概念混在了一起：

- Docker 容器内部端口。
- Docker 宿主机映射端口。
- 浏览器访问入口。
- Agent 连接入口。
- Agent 本地监听入口。

当部署始终是默认 9191 / 9192 且没有额外反代时，这些概念勉强可以看起来像一回事。

但一旦进入真实生产部署，尤其是以下场景：

- 端口冲突。
- 多项目并存。
- oneinstack 或 Nginx 反代到 443。
- Web 入口和 Agent 入口拆开。
- Agent 本地监听端口需要避让。

这些概念就必须分开建模，否则自动生成的命令一定会失真。

## 对后续改造的建议目标

后续改造建议至少满足以下目标：

- 后台生成命令时，不再假设固定是 9191 和 9192。
- 浏览器入口和 Agent 入口可以分别配置。
- 反代到 443 的面板入口，能够正确出现在脚本下载地址里。
- Agent 运行时真正要连接的 host 和 port，能够单独配置。
- `docker_command_with_listen` 的本地监听端口不再写死。
- 不改配置时，仍兼容现有默认部署方式。

## 推荐改法

## 方案一：先做最小闭环修复，优先推荐

这是最适合先落地的方案。

思路是：

- 不先大改 UI。
- 先在后端引入“公共入口配置”。
- 让部署命令生成逻辑改为读取这些配置。

建议新增配置项，至少包括：

- `NSR_PUBLIC_BASE_URL`
- `NSR_PUBLIC_AGENT_HOST`
- `NSR_PUBLIC_AGENT_PORT`
- `NSR_PUBLIC_AGENT_LISTEN_PORT_DEFAULT`

建议语义如下：

- `NSR_PUBLIC_BASE_URL`：给浏览器和安装脚本下载使用的公共入口，例如 `https://network-status.wingsrabbit.com`
- `NSR_PUBLIC_AGENT_HOST`：给 Agent 连接中心使用的 host，例如 `43.133.15.207`
- `NSR_PUBLIC_AGENT_PORT`：给 Agent 连接中心使用的端口，例如 `9194`
- `NSR_PUBLIC_AGENT_LISTEN_PORT_DEFAULT`：用于生成 `docker_command_with_listen` 时的默认本地监听端口，避免强写 9192

这种做法的优点：

- 改动集中。
- 对现有结构侵入小。
- 能优先解决“端口变了但命令不变”的主要矛盾。
- 可以兼容“前端入口和 Agent 入口不同”的真实生产场景。

推荐生成逻辑改为：

1. 安装脚本地址优先使用 `NSR_PUBLIC_BASE_URL`。
2. Agent 连接 host 优先使用 `NSR_PUBLIC_AGENT_HOST`。
3. Agent 连接 port 优先使用 `NSR_PUBLIC_AGENT_PORT`。
4. 本地监听端口优先使用 `NSR_PUBLIC_AGENT_LISTEN_PORT_DEFAULT`。
5. 如果这些配置都没有设置，再回退到默认 9191 / 9192 行为。

## 方案二：进一步做成“系统设置”，适合后续长期演进

如果希望管理员直接在后台配置，而不是改环境变量，可以继续往前走一步：

- 在系统设置里增加“公共访问配置”。
- 将这些配置存到数据库。
- `deploy-command` 接口优先读系统设置，再读环境变量，再读默认值。

建议至少提供以下设置项：

- 面板公共访问 URL
- Agent 公共访问 Host
- Agent 公共访问 Port
- Agent 默认监听 Port

这个方案更适合长期，但不一定需要第一步就做。

如果只是先解决“端口冲突时自动生成命令错误”这个核心问题，方案一已经足够。

## 如果未来目标是让 Agent 也能走 443 / HTTPS

这件事需要单独看，不建议和这次修复混成一个改动。

原因是当前 Agent 端只有：

- `--server`
- `--port`

并且最终被拼成：

- `http://{server}:{port}`

如果未来希望 Agent 也通过域名 443 接入，建议单独评估以下变更：

- 给 Agent 增加 `--scheme` 参数。
- 或直接支持 `--url` / `CENTER_URL` 这种完整地址形式。
- `server_url` 不再写死 `http://`。
- 明确 WSS / HTTPS 场景下的证书校验策略。

这一层不一定是当前必须做的，但必须在设计上意识到：

- 只改部署命令模板，不改 Agent URL 表达能力，无法完整覆盖所有 443 / TLS 场景。

## 推荐的修改文件范围

如果按“最小闭环修复”做，建议优先关注这些文件：

- server/api/nodes.py
- install.sh
- scripts/install-agent.sh
- README.md

如果要把配置能力做得更规范，建议额外引入：

- server/config.py 或等价配置读取层
- 系统设置相关后端接口与数据存储
- 前端系统设置页面

## 建议的修改原则

后续实现时建议坚持以下原则：

- 不要再在多个位置重复写死 9191 / 9192。
- 不要让前端自己拼命令字符串。
- 命令生成逻辑必须后端集中维护。
- 浏览器入口和 Agent 入口必须分开建模。
- 本地监听端口不要和中心连接端口耦合。
- 不配置新参数时，默认行为必须保持兼容。

## 建议验收口径

后续改完后，至少应该满足以下验收条件：

### 场景一：默认部署

- 不设置任何新配置。
- Web 仍走 9191。
- Agent 仍走 9192。
- 后台生成命令与当前默认部署一致。

### 场景二：只改宿主机端口

- 设置 `NSR_WEB_PORT=9193`。
- 设置 `NSR_AGENT_PORT=9194`。
- 后台生成命令不再出现 9191 / 9192，而应使用 9193 / 9194 或与之对应的公共配置。

### 场景三：浏览器入口走 443 域名，Agent 入口走单独端口

- 面板公共访问地址为 `https://network-status.wingsrabbit.com`。
- Agent 实际连接入口为 `43.133.15.207:9194`。
- 后台生成命令中的安装脚本下载地址应来自 443 域名。
- 后台生成命令中的 Agent 连接参数应是 `43.133.15.207:9194`。

### 场景四：带监听端口的 Docker 命令

- `docker_command_with_listen` 不再强行写死 `--listen-port 9192`。
- 若配置了默认监听端口，则使用配置值。
- 若没有配置，则明确展示默认值来源，或者干脆不生成带监听端口的命令。

### 场景五：安装脚本输出与健康检查

- `install.sh` 的健康检查不能再固定 `localhost:9191`。
- 安装完成后的提示信息，必须与当前实际端口一致。

## 对接手修改者的明确提醒

这个问题不是“改一个前端字符串”就能解决的。

至少要同时考虑以下几个层次：

- 后端命令生成接口。
- 一键安装脚本输出。
- Agent 安装脚本的下载入口与运行入口。
- 文档中的默认端口表述。
- 未来是否需要支持 HTTPS / 443 Agent 接入。

如果只改其中一个点，例如：

- 只改前端展示。
- 只改 README。
- 只改 `deploy-command` 的某个字符串。

那么系统很可能在下一个“端口冲突 + 反代入口”场景下再次出错。

## 当前阶段的结论

当前现象的根因已经非常明确：

- 应用实际部署端口是可变的。
- 但自动生成 Agent 部署命令的逻辑仍然是固定端口思维。
- 同时系统没有明确区分“浏览器入口”和“Agent 入口”。

因此，后续正确方向不是继续人工改命令，而是：

- 给系统补齐“公共入口配置”的建模。
- 让后端统一按真实入口生成部署命令。
