# 协作工作流程

本文档描述人与 AI Agent 协作完成一次版本更新的标准流程。

---

## 第一步：接收任务 & 问题确认

- 用户提出需求或报告 Bug，通常附带截图、日志等现场证据。
- Agent 根据描述定位问题范围，必要时向用户提问澄清意图。
- 确认本次更新的目标清单（修什么、加什么、改什么）。

> **踩坑备忘**
> - 截图比文字描述可靠得多。例如 v0.135 中"重置统计无效"，从截图能直接看到 SNT=262 没变，比口述"按了没反应"信息量大。
> - 用户说的"不能是我点进详情才开始"这类表述，要追问清楚是"任务创建时就应该在跑"还是"页面打开时看到已有的累计数据"——两者实现完全不同。

## 第二步：分析 & 定位

- 阅读相关源码，追溯调用链，理解现有逻辑。
- 确定 Bug 根因或功能缺口。
- 制定修改方案（改哪些文件、改什么、为什么）。

> **踩坑备忘**
> - 这一步最容易犯"只看到表象就动手"的错误。v0.135 的重置 Bug，表面是"点了没用"，实际根因是 fetchData 的自动刷新在 5 秒后把全部历史数据重新合并回来了——如果只盯 resetMtrStats 函数，永远找不到问题。
> - 跨组件调用链（前端 → WebSocket → 后端 → Agent → 后端 → 前端）需要完整走一遍，缺任何一环都会导致功能断裂。
> - 外部工具的参数语义不要想当然。curl 的 timing 字段是累计值而非增量；mtr 的 `-w` 和 `--json` 互斥；Python 函数参数即使看起来有默认值也必须显式传。

## 第三步：实现 & 修改代码

- 按方案逐一修改代码（前端 / 后端 / Agent 各层）。
- 每改一处保持可编译状态，避免一次性大改导致难以排查。

> **踩坑备忘**
> - 前端状态管理是重灾区。v0.135 中 `mtrStartTime` 被设为 null 后触发了"首次加载"分支重新拉全量数据——修改 ref/state 前务必搜索所有读取该值的地方，画出状态流转。
> - InfluxDB 查询对类型敏感：`mean()` 不能用在 boolean 字段上（报 400）；聚合后的 `success` 字段变成 float 0.0，`=== false` 匹配不上。
> - PowerShell 远程执行 SSH 命令时，`$host` 是只读变量，nohup 的重定向符号会被 PowerShell 解析——用单引号包裹整条远程命令。

## 第四步：版本号更新

- `version.py`（后端 + Agent）和 `web/package.json`（前端）同步更新版本号。
- 两处版本号必须一致。

> **踩坑备忘**
> - 容易漏改 `web/package.json`，导致前端显示旧版本号。改完后可以在构建产物里 grep 确认（如搜索 `const o="0.135.0"`）。
> - 版本号更新应在部署前完成，这样部署上去的就是新版本，验证时可以直接在页面确认版本号正确。

## 第五步：编写更新日志

- 在 `project-files/update/` 下创建 `update_vX.XXX.md`。
- 内容包括：日期、Bug 修复列表、新增功能、验证结果、版本变更记录。

> **踩坑备忘**
> - 先写日志再部署验证，好处是写的时候记忆最清晰，验证通过后只需补充验证结果数据即可。
> - 更新日志面向"下次接手的人（包括未来的 AI Agent）"，重点写"改了什么、为什么这么改"，而非贴大段代码。

## 第六步：部署

- 前端：`docker compose build --no-cache backend`（多阶段构建含前端）→ `docker compose up -d`。
- 后端热补丁（小改动）：`docker cp` 文件到容器 → `docker restart ns-center`。
- Agent：SCP 从本地 Windows 推送到各节点 → `systemctl restart networkstatus-agent`。
- 请读取文档下 testread 文件夹，以进行远程部署。
> **踩坑备忘**
> - Agent 部署源是本地 Windows 机器（SCP + SSH key），不是从服务器拉取。
> - Docker 构建缓存可能导致旧代码残留，前端务必加 `--no-cache`。
> - 部署顺序：后端 → Agent → 前端，确保 API 兼容性。如果后端有新的 WebSocket 事件，Agent 必须先更新才能响应。

## 第七步：验证

- 通过 Web 界面 / API / 日志确认功能正常。
- 对本次改动涉及的所有协议、场景逐一测试。
- 验证版本号在页面上显示正确。
- 用户确认无误后进入下一步。

> **踩坑备忘**
> - 不要只测"改了的东西"，也要回归测试相邻功能。例如改了 MTR 重置，也要确认 ICMP/TCP/UDP 三种协议都还能正常出数据。
> - 前端构建产物的 hash 变化可以作为"确实重新构建了"的快速验证手段。
> - InfluxDB 数据有延迟，刚部署完可能需要等一个探测周期再看结果。

## 第八步：提交 & 推送

- `git add` → `git commit`（附简明 commit message）→ `git push origin NetworkStatus-Rabbit-NG`。
- 清理不需要的分支（如临时 copilot 分支）。

> **踩坑备忘**
> - commit message 写明版本号和主要改动摘要，方便 `git log --oneline` 快速回溯。
> - 推送前确认当前分支是 `NetworkStatus-Rabbit-NG`，不要误推到 master。
> - 删除远程分支属于不可逆操作，先向用户确认再执行。

---

## 注意事项

- 涉及共享环境的不可逆操作（删分支、force push 等）先向用户确认。
- 每一步完成后向用户简要汇报，而非闷头做完全部再说。
- 如果改动跨多个组件（前端 + 后端 + Agent），部署顺序通常为：后端 → Agent → 前端，确保 API 兼容。
