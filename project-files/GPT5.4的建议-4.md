# GPT5.4 的建议 4

## 一、先说结论

我重新通读了当前这版 v1.3 的 project.md。

这次我继续使用同一套评审口径：

**不是看“人能不能大概照着做”，而是看“这份文档是否已经足够一致、足够少歧义，可以直接交给 AI Agent 开工，而且不太需要 AI 自己补关键设计”。**

在这个标准下，我这次的判断是：

**这版文档我给 9.6 / 10。**

我愿意把它提到 9.6，原因很明确：

1. 它已经不只是“可以交给 AI Agent 开工”
2. 而且已经接近“AI Agent 按文档做，大方向和大部分边界都不会跑偏”

也就是说，如果现在把它交给 AI Agent 去实现项目主链路，我认为：

1. 主协议不会乱
2. 数据模型不会乱
3. 权限边界不会乱
4. 前后端接口不会大面积各写各的

这已经非常接近你想要的状态了。

但我这次仍然没有直接给到 9.8 或 10，原因不再是“缺内容”，而是：

**文档内部还残留了几处规格不一致。它们每一处单看都不大，但恰好都属于 AI Agent 最容易被误导的类型。**

换句话说：

现在最大的风险已经不是“写不出来”，而是“AI 会严格照文档写，却不知道应该信哪一处定义”。

## 二、这次为什么能从 9.4 提到 9.6

这版最值得肯定的是，它把上一轮里最关键的三个问题都真的补掉了。

### 1. Cookie 与 WebSocket 认证冲突已经修掉

上一轮我认为最严重的问题，是 Cookie `Path=/api` 和 WebSocket 认证共用 Cookie 这两条彼此冲突。

这次已经明确改成：

1. Cookie `Path=/`
2. REST API 和 WebSocket 共用认证 Cookie
3. 文档补了“前后端同站点部署”的前提

这让 AI Agent 不再需要自己拍脑袋决定认证路径。

### 2. WebSocket 失败语义终于写进规格了

7.9 的加入很关键。

现在至少已经写清楚了：

1. 握手失败怎么处理
2. 已连接状态下的 Token 过期怎么处理
3. 权限不足怎么处理
4. 无效订阅怎么处理
5. 哪些情况断连，哪些情况不断连

这一层补上后，Socket.IO 相关实现的自由发挥空间明显缩小了。

### 3. 剩余高风险 REST 接口示例已经补了大半

这一版补进来的高风险接口示例是有效的：

1. 用户角色修改
2. Webhook 通道
3. 告警历史
4. 系统设置
5. 任务启停

这意味着现在的 REST API 设计已经基本从“说明文”进入“可直接照着写”的状态。

### 4. 自测清单已经开始覆盖失败场景

17 章现在不只测成功路径，也开始测：

1. WebSocket 鉴权失败
2. readonly 权限越界
3. 删除最后一个 admin
4. 无效订阅
5. 数据边界输入

这一点很重要，因为它让 AI Agent 不容易只把 happy path 做通就结束。

## 三、为什么我现在还不给更高分

这次不到更高分，已经不是因为文档少了什么模块，而是因为文档内部还存在几处**规格互相打架**的情况。

这是现在最需要警惕的阶段，因为：

1. 内容越完整
2. 章节越多
3. 越容易出现“局部正确，但全局不一致”

而这正是 AI Agent 最怕的文档形态。

### 1. REST 错误规范和接口示例的错误结构不一致

这是我这轮看到的最明确的内部冲突之一。

14.9 已经明确定义：

1. 所有非 2xx 响应统一返回
2. 格式为

```json
{
  "error": {
    "code": 422,
    "type": "validation_error",
    "message": "...",
    "details": {}
  }
}
```

而 14.10 的高风险接口错误示例却写成了：

```json
{ "error": { "code": "PERMISSION_DENIED", "message": "不允许通过 Web 降级 admin 角色" } }
```

这里至少有两层冲突：

1. 14.9 规定 `error.code` 是 int，示例却用了 string
2. 14.9 规定有 `type`，示例却没有

这类冲突对 AI Agent 的伤害很直接：

1. 它会不知道该信规范表还是示例
2. 后端和前端可能分别照不同部分实现

这个问题不大，但一定要修，因为它已经属于“照文档严格实现也可能做错”的级别。

### 2. WebSocket 错误格式和 REST 统一错误格式也没有真正统一

7.9 开头写的是：

**“Socket.IO 错误事件统一格式，与 REST API 错误格式保持一致”**

但后面给出的示例是：

```json
{
  "code": "WS_AUTH_FAILED",
  "message": "Cookie 中的 JWT 无效或已过期"
}
```

这和 14.9 的 REST 格式并不一致，因为 REST 是：

```json
{
  "error": {
    "code": 401,
    "type": "auth_error",
    "message": "..."
  }
}
```

所以现在的问题不是 WebSocket 设计有问题，而是文档说“保持一致”，但示例并没有做到一致。

建议二选一明确：

1. 要么承认 WS 与 REST 是两套不同错误格式
2. 要么把 WS 也改成 `error` 包裹结构

现在这种“口头上统一，结构上不统一”的状态，最容易让 AI 实现出半统一版本。

### 3. WebSocket 路径表述仍然没有完全统一

我看到文档里现在至少存在三种相关路径描述：

1. 5.1 中写的是反代 `/ws/* → Flask Socket.IO`
2. 7.3 连接生命周期写的是 `ws://center:port/ws/agent`
3. 9.5 和 7.9 又写的是 WebSocket 握手路径 `/socket.io/*`

这里的问题不是哪个一定错，而是：

**AI Agent 不知道最终标准到底是哪一个。**

如果你是想表达：

1. Nginx 外层路由是 `/ws/*`
2. Socket.IO 内部路径是 `/socket.io/*`

那就需要写得更硬、更明确。

否则 AI 很可能会：

1. 一部分代码按 `/ws`
2. 一部分代码按 `/socket.io`
3. 最后自己再补一层路径适配

这类问题属于典型的“不是复杂问题，但很容易搞得很烦”的问题。

### 4. 告警历史表结构与 API 示例字段名不完全一致

6.3 的 `alert_history` 表里定义的是：

1. `rule_type`
2. `value`
3. `threshold`
4. `webhook_sent`

而 14.10 的 `GET /api/alerts/history` 返回示例里却用了：

1. `metric`
2. `actual_value`
3. `threshold`
4. `notified`

这不一定是错误，因为“数据库字段”和“API DTO 字段”可以不同。

但问题在于：

**文档目前没有明确说明这里是“内部表字段”和“对外返回字段”的刻意映射”。**

AI Agent 很可能会因此自己决定：

1. API 直接透出数据库字段
2. 或者另做映射
3. 或者前后端各自理解不同字段名

如果你希望字段不同，那最好明确写一句：

1. API 可对 DB 字段做语义化映射
2. 哪些字段是一一对应

不然这仍然会制造额外解释空间。

## 四、我建议这次优先补的 3 个点

如果只做最少改动，我建议优先修这 3 个地方。它们不是“大改”，但能明显提升文档最终一致性。

### 1. 统一错误响应里的 `code` / `type` 规范

建议把 14.9 和 14.10 一次性统一掉，明确写死：

1. `error.code` 到底是 HTTP 数字码，还是字符串业务码
2. `error.type` 是否必须存在
3. 示例必须完全跟规范一致

我更建议：

1. `error.code` 保持 HTTP 数字码
2. `error.type` 放字符串业务类型

例如：

```json
{
  "error": {
    "code": 403,
    "type": "permission_denied",
    "message": "不允许通过 Web 降级 admin 角色"
  }
}
```

这样最稳，也最不容易让 AI 搞混。

### 2. 把 WebSocket 路径统一成单一标准

建议文档里明确写成一句硬规则，例如：

1. 外部统一暴露路径就是 `/socket.io/`
2. Nginx 只负责反代这个路径
3. 文档中不再出现 `/ws/agent` 这种并行说法

或者反过来统一成 `/ws/socket.io`，也可以。

重点不是选哪个，而是：

**只能保留一个最终标准。**

### 3. 明确 API DTO 与数据库字段是否允许映射

尤其是 `alert_history` 这类地方，建议补一句规则：

1. API 返回字段可以与数据库字段不同，但必须在文档中列出映射关系

或者更简单：

1. 直接把 API 示例字段名改到和表结构一致

两种都可以，关键是不要让 AI 自己决定。

## 五、这次新发现的 3 个次级问题

这些不是主要阻塞，但值得作为最后一轮打磨项保留。

### 1. 13.2 的“MVP 权衡说明”是好的，但还可以更硬一点

现在已经写清楚“命令行明文 Token”是 MVP 权衡，这很好。

如果想更进一步，我建议补一句：

1. 初版允许此方案
2. 但 AI Agent 不应自行扩展为明文写日志、明文回显、或在 UI 再次展示 Token

这样可以把安全边界再压实一点。

### 2. Dashboard 筛选参数已经定义，但排序规则还没单列

现在 14.4 已经有筛选参数，这很好。

如果再补一句：

1. dashboard 默认按什么排序
2. 是否允许前端传排序字段

那前后端就会更容易完全对齐。

### 3. AI 自测清单已经很强，但可以再补“文档一致性自查”

现在的自测清单更多是行为测试。

如果目标是交给 AI Agent，我甚至建议再加 2 到 3 条“规格一致性自查”：

1. REST 错误示例是否与统一错误规范一致
2. WS 错误示例是否与 WS 错误规范一致
3. API 返回字段是否与文档字段映射一致

这会非常贴近你现在这个阶段的真实需求。

## 六、如果只做最少改动，我建议按这个顺序补

### 第一优先级

1. 统一 REST 错误规范和错误示例
2. 统一 WebSocket 路径标准

### 第二优先级

3. 明确 API 字段与数据库字段的映射规则

### 第三优先级

4. 补 Dashboard 排序规则
5. 给自测清单补文档一致性自查项

如果只补第一优先级，我认为这份文档已经非常接近 9.8。

如果第一、第二优先级都补掉，我基本就愿意把它视为“可以放心直接交给 AI Agent 开工”的高完成度规格文档。

## 七、最终结论

我的最终结论是：

**v1.3 这版 project.md 已经非常接近最终可交付状态，我给 9.6 / 10。**

它现在最大的优点是：

1. 不是缺模块
2. 不是缺协议
3. 不是缺状态机
4. 而是已经开始进入“只剩统一口径和规格一致性”的阶段

这其实是一个很好的信号，因为说明：

**它已经基本跨过了“AI Agent 会不会做错方向”的阶段，开始进入“AI Agent 能不能毫无歧义地严格按文档实现”的最后打磨阶段。**

现在还没到更高分的核心原因只有一句话：

**文档内部还有少量格式和命名上的不一致，会让 AI Agent 不知道到底该信规范表、字段表，还是示例。**

如果下一版把这些一致性问题再收一轮，我认为它就已经非常接近你真正想要的目标：

**可以放心直接交给 AI Agent 开工，而且 AI 基本不需要自己补关键设计，也不太会被文档内部冲突误导。**
