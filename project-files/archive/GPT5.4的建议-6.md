## 二、为什么这次已经几乎到 10 分

v1.5 把我上一轮点名的 4 个问题，基本都真正修掉了，而且不是只写在变更记录里，正文也跟着统一了。

### 1. TCP Probe 的自检逻辑已经回到可实现状态

上一轮最大的实现级问题，是 TCP 自检要求“连接自身 WS 端口”，但 Agent 并没有定义本地监听端口。

这次已经改成：

1. 检查 Python `socket` 模块可导入
2. `socket.create_connection()` 可正常调用
3. 自检失败不阻塞 Agent 启动

这就对了。

这意味着 AI Agent 不会再被一个逻辑上不成立的自检规则带偏。

### 2. connect_error 的读取方式已经统一

上一轮 7.9 最大的问题，是流程说明和前端示例不一致：

1. 一处说从 `connect_error.data` 取
2. 一处又在示例里读 `err.message`

这次已经收敛成单一规则：

1. 握手拒绝时后端通过 `data` 传错误载荷
2. 前端统一读取 `err.data.code`
3. `err.message` 不再作为业务判断依据

这会显著降低 AI Agent 在前端实现时自己发明兼容逻辑的概率。

### 3. Token 的安全措辞已经收紧到合适程度

这次 13.2 的表达比上一轮更准确：

1. 允许命令行明文传参是 MVP 权衡
2. 不是默认最佳实践
3. 明确禁止日志、API、UI 回显 Token

这一点很重要，因为 AI Agent 很容易把文档里的解释语言当成“隐性推荐方案”。

现在这个风险已经基本被拿掉了。

### 4. 告警历史接口的“字段完整性”也补上了

现在 `GET /api/alerts/history` 已经明确：

1. 示例展示的是完整字段集合
2. 前端依赖哪些字段被写死

这能避免 AI 把示例误读成“随便返回几个字段也行”。

从 AI Agent 落地角度看，这是一个很有价值的小修复。

## 三、为什么我这次还是没有直接给 10 分

现在剩下的问题只有一个，而且已经不是设计问题，而是**最后一处验收表达没有完全收平**。

### 17 章里还有两条 422 自测断言没把 `type` 写全

在“失败场景自测 → 数据边界”下面，目前是：

1. 探测间隔设为 0 或 61 → `422 + validation_error`
2. 创建任务时 `target_type=internal` 但 `target_node_id` 为空 → `422`
3. 创建任务时 `target_type=external` 但 `target_address` 为空 → `422`

这里的问题非常小，但从严格规格角度看，它确实还没完全一致。

因为 14.9 已经明确规定：

1. HTTP 状态码只是 `error.code`
2. 业务错误类型还要有 `error.type`
3. 自测断言最好和正式规格完全同构

所以这两条如果只写 `422`，AI Agent 在写自动化测试时，可能就不会去断言 `validation_error`，从而让测试口径比正文规格更松。

这不会导致项目实现跑偏，但它确实说明：

**文档还差最后两处文字级统一，才算真的完全收口。**
