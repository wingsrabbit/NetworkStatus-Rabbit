## 二、这次为什么能从 9.6 提到 9.8

这版最大的进步，是它把第四轮指出的“一致性问题”真正收掉了大半。

### 1. REST 错误规范和示例终于统一了

上一轮最大的问题之一，是 14.9 说一套，14.10 错误示例又写成另一套。

这次已经统一成：

1. `error.code` 为 HTTP 数字码
2. `error.type` 为字符串业务类型
3. 示例和规范保持一致

这会显著减少 AI Agent 在错误处理层面自己发明格式的空间。

### 2. WebSocket 路径已经基本收敛为单一标准

现在文档已经把之前并存的 `/ws/*`、`/ws/agent`、`/socket.io/*` 收敛到了 `/socket.io/*` 这一套标准上。

这件事看起来小，但对 AI Agent 很关键，因为路径一旦有多套说法，代码实现就很容易出双轨。

### 3. alert_history 字段名已经完成统一

上一轮里，表字段和 API 字段名并不一致。

现在这部分已经统一成：

1. `metric`
2. `actual_value`
3. `notified`

这意味着 AI Agent 在做模型、DTO、前端展示时，不再需要自己猜映射关系。

### 4. 文档已经开始包含“规格一致性自查”

这一点我认为非常好。

因为你的目标本来就不是“AI 能凑一版出来”，而是：

**AI 要尽量严格、机械、低自由发挥地把规格落地。**

17 章新增的“规格一致性自查”正是在朝这个目标靠。

## 三、这次我仍然不给 10 分的原因

现在的文档已经非常完整，所以我这次不再盯大块结构，而只盯那些仍然可能让 AI Agent 严格照文档写却写错的地方。

### 1. Agent 的 TCP 自检逻辑描述存在明显实现风险

这是我这轮看到的最值得指出的一条。

11.5 的能力发现里写：

1. TCP Ping 自检方式为“尝试 TCP 连接 `127.0.0.1:自身WS端口`”
2. 判定标准是 socket 连接成功

这里的问题是：

**Agent 本身是 WebSocket 客户端，不是本地 WebSocket 服务端。文档里也没有定义 Agent 在本地监听一个 TCP 端口。**

这意味着按当前描述严格实现时，AI Agent 会遇到一个很实际的问题：

1. 它要连哪个“自身WS端口”？
2. 这个端口如果根本不存在，那自检逻辑就是伪命题

这不是小问题，因为它会直接误导 Agent 能力发现的实现。

我建议把 TCP 自检方式改成下面两种之一：

1. 检查 Python `socket` 能力可用，并对一个已知可达目标执行一次短连接自检
2. 或更简单，定义 TCP Probe 插件自检为“模块可导入 + socket connect 能执行”，而不是连接不存在的本地端口

如果不改，AI Agent 很容易照着文档写出一个永远失败或逻辑不成立的 TCP 自检。

### 2. AI 自测清单里仍残留了旧版错误码表达

文档主体现在已经统一成：

1. REST `error.code` 是数字 HTTP 状态码
2. `error.type` 才是业务类型字符串

但 17 章的失败场景自测里，仍然还有类似写法：

1. `403 PERMISSION_DENIED`
2. `409 CONFLICT`
3. `422 VALIDATION_ERROR`

这会造成一个 subtle 但真实的问题：

1. 规格章节要求一套
2. 自测章节又像是在暗示另一套
3. AI Agent 在写测试时可能会按大写字符串断言错误码

也就是说，文档实现规格已经统一了，但验收规格还没完全跟上。

建议统一改成：

1. `403 + permission_error`
2. `409 + conflict`
3. `422 + validation_error`

这样主规格和自测规格才完全一致。

### 3. WebSocket 握手失败的错误载荷描述还有最后一点模糊

7.9 里写了：

1. 握手失败时触发 `connect_error`
2. 错误信息通过 `connect_error.data` 传递

但下面的前端示例又写成：

1. `err.message.includes('AUTH')`

这会造成一个小冲突：

1. AI Agent 不知道该从 `err.message` 取
2. 还是从 `err.data` 取

这已经不是协议大问题，但确实还是一个“实现接口层的小歧义”。

建议直接写死前端约定，例如：

1. `connect_error.message` 固定为 `WS_AUTH_FAILED`
2. 详细信息放在 `connect_error.data`

或者反过来统一成只读 `data.code`。

重点不是选哪种，而是：

**只能保留一种前端读取方式。**

### 4. “Token 明文传参是业界常见做法”这句话略显过宽

13.2 已经补上了安全边界，这很好。

但“这是容器化部署和运维脚本的业界常见做法”这句话还是稍微宽了一点，容易让 AI Agent把它理解成“这是推荐方案”。

更精确的说法建议是：

1. 初版为降低实现复杂度，接受命令行传参
2. 但这属于 MVP 权衡，不是默认最佳实践

本质上你已经表达了这个意思，只是可以再收紧一下措辞，避免 AI 用它给更宽松的安全实现背书。

## 四、我建议这次优先补的 3 个点

如果只做最少改动，我建议优先补下面这 3 个点。它们都是小改动，但收益非常高。

### 1. 先修正 TCP Probe 的能力发现自检方式

这是我这轮的第一优先级。

建议改成：

1. “TCP Probe 插件可导入且 `socket.create_connection()` 可正常执行”

或者：

1. “对管理员配置的已知测试目标进行一次短连接自检”

不要再保留“连接自身 WS 端口”这种不存在明确前提的说法。

### 2. 把 AI 自测清单里的错误码表达全部与 14.9 对齐

建议把所有类似：

1. `403 PERMISSION_DENIED`
2. `409 CONFLICT`
3. `422 VALIDATION_ERROR`

改成类似：

1. `403 + permission_error`
2. `409 + conflict`
3. `422 + validation_error`

这样测试断言就不会和正文规范打架。

### 3. 把 connect_error 的读取方式写成单一规则

建议补一句硬规则，例如：

1. 握手失败时前端只读取 `err.data.code`
2. `err.message` 不作为业务判断依据

或者反过来统一读 `message`，也可以。

关键是别两种都保留。

## 五、这次新发现的 2 个次级建议

这些不影响“直接开工”，但如果你还想继续追求文档极限完成度，它们值得保留。

### 1. 建议给能力发现增加“自检失败不影响 Agent 启动”的硬说明

当前文档里已经定义了能力发现，但没明确说：

1. 如果某协议自检失败，Agent 是否仍然正常启动
2. 还是直接启动失败

从产品逻辑看，显然应该是：

1. 某协议不可用只影响 capabilities
2. 不影响整个 Agent 启动

这条最好写死，因为 AI Agent 很可能会默认把自检失败当成启动失败。

### 2. 建议给告警历史响应补“字段是否为最小子集”说明

现在 `GET /api/alerts/history` 返回示例是合理的，但如果它只是最小字段子集，建议加一句：

1. 示例不是完整字段全集
2. 前端至少依赖哪些字段

否则 AI 可能会把示例当成唯一返回字段集合，从而在后续扩展时做得过窄。
