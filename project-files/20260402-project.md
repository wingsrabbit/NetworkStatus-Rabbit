## 3. 项目是什么

### 3.1 核心角色

#### Center

中心节点，负责：

1. Web UI。
2. REST API。
3. Agent WebSocket 接入。
4. InfluxDB 时序写入与查询。
5. 告警判定、告警历史、Webhook。
6. 节点、任务、用户、系统设置管理。

#### Agent

部署在被监控节点上的无头客户端，负责：

1. 连接 Center。
2. 上报本机能力信息。
3. 接收任务同步。
4. 定时执行探测。
5. 本地缓存未 ACK 数据。
6. 断线后批量补传。
7. 可选地开放 TCP+UDP echo 端口，供内部 TCP/UDP 任务探测。

#### Frontend

面向管理员或只读用户的监控界面，负责：

1. 仪表盘总览。
2. 任务详情图表。
3. 节点、任务、用户、告警、设置管理。
4. 接收 Dashboard 命名空间的实时推送。

## 6. Agent 当前真实实现

### 6.1 启动流程

`agent/main.py` 的真实流程是：

1. 解析命令行参数。
2. 如果传了 `--listen-port`，启动 `EchoServer`，同时监听 TCP 和 UDP。
3. 初始化 `LocalCache`。
4. 初始化 `TaskScheduler`。
5. 初始化 `WSClient`。
6. 启动后台清理线程，每小时清理超过 3 天的已 ACK 缓存。
7. 进入无限重连循环：连接 Center，等待断开，再重试。

### 6.2 Agent 参数

`agent/config.py` 当前支持：

| 参数 | 作用 |
| --- | --- |
| `--server` | Center 地址 |
| `--port` | Agent 通道端口，默认 9192 |
| `--node-id` | 节点 ID |
| `--token` | 节点 token |
| `--data-dir` | 本地缓存目录 |
| `--listen-port` | 可选监听端口，开启后可作为内部 TCP/UDP 目标 |

### 6.3 Agent 能力发现

连接成功后，`agent/ws_client.py` 会在 `/agent` 命名空间发 `agent:auth`，并带上 `capabilities`：

1. 支持的协议列表。
2. 不支持的协议列表。
3. 不支持原因。
4. Agent 版本。
5. `listen_port`。
6. OS 信息。
7. 自动探测到的公网/私网 IP。

能力发现来自各 probe 的 `self_test()`。

当前各协议自检方式：

| 协议 | 自检逻辑 |
| --- | --- |
| ICMP | 真实执行本机 `ping 127.0.0.1` |
| TCP | 使用 Python `socket.create_connection()` 做最小可用性验证 |
| UDP | 创建 Python UDP socket |
| HTTP | 检查 `curl`，若无则退回 `requests` 模块是否可导入 |
| DNS | 检查 `dig` 或 `nslookup` |
| MTR | 检查 `mtr --version` |

### 6.4 任务调度

`agent/scheduler.py` 的真实行为：

1. 每个任务独立线程。
2. 按固定节奏执行，不是简单 `sleep(interval)`。
3. 若一次探测超时导致超过周期，会重置节奏，避免连续堆积。
4. 任务来源只有 Center 同步，不读本地任务文件。

### 6.5 本地缓存

`agent/local_cache.py` 使用 SQLite 表 `local_results`：

| 列 | 作用 |
| --- | --- |
| `result_id` | 全局唯一结果 ID |
| `task_id` | 任务 ID |
| `payload_json` | 完整结果 JSON |
| `ack_status` | `pending/sent/acked` |
| `batch_id` | 批量补传批次号 |
| `retry_count` | 批量补传重试次数 |
| `created_at` | 创建时间 |
| `sent_at` | 最近发送时间 |
| `acked_at` | 确认时间 |

真实生命周期：

1. 探测结果先写本地缓存，状态 `pending`。
2. 如果当前已连通，立即发送并标记 `sent`。
3. 收到 `center:result_ack` 或 `center:batch_ack` 后标记 `acked`。
4. 每小时清理超过 3 天的已 ACK 记录。

### 6.6 断线补传

`agent/ws_client.py` 的 `_backfill()` 逻辑：

1. 取出所有未 ACK 记录。
2. 每 100 条切一批。
3. 写入 `batch_id` 并递增 `retry_count`。
4. 发送 `agent:probe_batch`。

这说明历史 bug 报告里关于“batch_id / retry_count 只是占位”的问题，在当前代码里已经修掉。

### 6.7 断开连接时的行为

Agent 断线后会：

1. 停止心跳线程。
2. `scheduler.stop_all()`，停止所有探测任务。
3. 本地缓存继续保留。
4. 重连后强制把本地 `config_version` 置 0，从而触发全量同步。

## 7. 协议与探测实现矩阵

### 7.1 当前支持的协议

当前代码支持：

1. `icmp`
2. `tcp`
3. `udp`
4. `http`
5. `dns`
6. `mtr_icmp`
7. `mtr_tcp`
8. `mtr_udp`

### 7.2 每种协议的真实实现来源

| 协议 | 适配层 | 底层实现 | 输出字段 |
| --- | --- | --- | --- |
| ICMP | `agent/probes/icmp_probe.py` | `agent.tools.icmp_ping.monitor_ping` | `latency`、`packet_loss` |
| TCP | `agent/probes/tcp_probe.py` | `agent.tools.tcp_ping.monitor_tcp_ping` | `latency`、`tcp_time` |
| UDP | `agent/probes/udp_probe.py` | `agent.tools.udp_ping.ping_udp` | `latency`、`packet_loss`、`jitter` |
| HTTP | `agent/probes/http_probe.py` | `agent.tools.curl_ping.monitor_curl` | `status_code`、`dns_time`、`tcp_time`、`ttfb`、`total_time` |
| DNS | `agent/probes/dns_probe.py` | `agent.tools.dns_lookup.monitor_dns` | `latency`、`dns_time`、`resolved_ip` |
| MTR | `agent/probes/mtr_probe.py` | `agent.tools.mtr.monitor_mtr` | `latency`、`packet_loss`、`jitter`、`hops`、`extra` |

### 7.3 agent/tools/ 的定位

`agent/tools/` 目录中包含：

1. 被集成进仓库的底层网络工具代码。
2. 这些工具自己的 README 和分析脚本。

要点：

1. 运行时真正调用的是各协议的 `monitor_*` 或核心函数。
2. `analyze_*` 脚本、子目录 README、LICENSE 主要是随工具一并保留的说明与离线分析工具，不在平台运行主链路中被直接调用。
3. `agent/tools/mtr/monitor_mtr.py` 是本仓库新增的 MTR 包装器，不属于原来的五协议工具集合。

### 7.4 内部 TCP/UDP 任务的目标端口逻辑

对于内部目标：

1. 若任务是 TCP 或 UDP，服务端会优先从目标节点 `capabilities.listen_port` 中取端口。
2. 也就是说，目标 Agent 必须以 `--listen-port` 启动，才适合当内部 TCP/UDP 探测目标。
3. 如果目标节点没开放 `listen_port`，任务仍可创建，但实际探测很可能失败。

### 7.5 MTR 的特殊性

MTR 的结果不仅有最终 hop 指标，还包含：

1. `hops`：逐跳信息数组。
2. `extra.mtr_src`：MTR 识别到的源。
3. `extra.mtr_dst`：MTR 识别到的目的。

这些字段会：

1. 写入 InfluxDB。
2. 实时推给详情页。
3. 在前端以终端风格表格展示。

## 8. Center 当前真实实现

### 8.1 Flask 应用工厂

`server/app.py` 当前做了：

1. 加载配置。
2. 初始化 `db`、`jwt`、`socketio`。
3. 初始化 `influx_service`。
4. 注册所有蓝图。
5. 注册 WS 命名空间处理器。
6. `db.create_all()`。
7. 注册统一错误处理器。
8. 启动 3 个后台任务：心跳检查、快照推送、同步重试。

### 8.2 后台任务

#### 心跳检查器

每 10 秒运行一次，负责：

1. 根据最近 120 秒内心跳数判断 online/offline。
2. 在线节点更新 `last_seen`。
3. 状态变化时推送 `dashboard:node_status`。
4. 节点 offline/online 时推送一条 dashboard alert 类型消息。

#### 仪表盘快照推送器

每秒运行一次，负责：

1. 从内存取每个任务最新结果。
2. 读取当前告警状态。
3. 向 `/dashboard` 广播 `dashboard:probe_snapshot`。

#### 任务同步重试器

每 10 秒运行一次，负责：

1. 检查哪些节点仍处于 `pending` 同步状态。
2. 超时后重发 `center:task_sync`。
3. 超过最大重试后标记为 `desync`。

### 8.3 节点状态服务

`server/services/node_service.py` 的规则是硬编码的：

1. 心跳窗口：120 秒。
2. 离线阈值：窗口内 `<= 20` 个心跳即离线。
3. 在线判定：窗口内 `> 20` 个心跳。

当前这些常量没有读系统设置表。

### 8.4 任务同步状态机

`server/services/task_service.py` 当前维护：

1. `pending`
2. `synced`
3. `desync`

规则：

1. 任务增删改或启停时，源节点 `config_version + 1`。
2. 然后 `mark_sync_pending()`。
3. Agent 收到任务事件后回 `agent:task_ack`。
4. 超时 30 秒未 ACK，重试。
5. 最多 3 次，仍失败则进入 `desync`。

### 8.5 Agent 认证

`server/ws/agent_handler.py` 的认证条件：

1. `node_id` 存在。
2. 节点已创建。
3. 节点 `enabled=True`。
4. `token` 通过 bcrypt 验证。

认证成功后，会更新：

1. `status=online`
2. `last_seen`
3. `agent_version`
4. `public_ip`
5. `private_ip`
6. `capabilities`

### 8.6 Probe 结果处理链路

单条结果到达 Center 后，真实处理顺序是：

1. 验证任务是否存在。
2. 验证结果是否由该任务所属源节点上报。
3. 使用 `result_id` 做去重检查。
4. 写入 InfluxDB。
5. 标记 dedup 已写入。
6. 回 `center:result_ack`。
7. 更新内存最新结果缓存。
8. 推给已订阅该任务详情页的 Dashboard 客户端。
9. 若结果时间戳在 60 秒内，进入告警评估。

批量补传与此类似，但只写入、不参与告警。

### 8.7 去重实现

当前去重不是单纯内存缓存，而是“双层去重”：

1. 内存 `OrderedDict`，10 分钟 TTL，用于性能优化。
2. `data/dedup.db` 里的 `written_results` 表，用于持久去重，保留 7 天。

这说明早期 bug 文档里“只做了 10 分钟内存去重”的问题，在当前代码中已经修复。

## 9. 告警当前真实实现

### 9.1 告警状态机

`server/services/alert_service.py` 当前实现了三态：

1. `normal`
2. `alerting`
3. `recovering`

规则：

1. 基于滑动窗口判断是否超阈值。
2. 支持冷却期。
3. 连续恢复正常达到阈值后发送 recovery。

### 9.2 支持的告警指标

当前实现支持：

1. `latency`
2. `packet_loss`
3. `continuous_fail`

其中 `continuous_fail` 通过 `success` 字段累计失败次数实现。

### 9.3 告警落库

告警并不是只停留在内存：

1. 会写入 `alert_history` 表。
2. 会记录 `message`。
3. 会记录 `alert_started_at`。
4. recovery 时会计算并写入 `duration_seconds`。
5. 会向启用的 webhook 通道广播。
6. 会向 Dashboard 广播 `dashboard:alert`。

这说明旧的 bug 报告里“告警历史不落库”的问题，当前代码也已经修复。

### 9.4 告警历史接口

当前有两条读取路径：

## 14. 部署与运维

### 14.1 Center 安装方式

有两条路：

#### 路线 A：根目录 `install.sh`

支持：

1. 服务端模式。
2. Agent 模式。

服务端模式会：

1. 安装 Docker。
2. clone 仓库到 `/opt/NetworkStatus-Rabbit`。
3. 自动生成 `.env`。
4. `docker compose up -d --build`。

#### 路线 B：手动 compose

1. 准备 `.env`
2. `docker compose up -d --build`

### 14.2 Center 容器启动时真实动作

`scripts/entrypoint.sh` 会：

1. 把前端构建产物复制到共享卷。
2. 等 InfluxDB 准备好。
3. 调 `setup-influxdb.py` 初始化 bucket 和 downsampling task。
4. 创建 SQLite 表。
5. 运行 v0.133 的 SQLite 迁移：补 `token_plain` 列。
6. 若没有用户，创建默认 `admin/admin123456`。
7. 启动 Flask + Socket.IO。

### 14.3 Agent 安装方式

当前实际有两种完全不同的安装链路：

#### 链路 A：根目录 `install.sh agent`

特点：

1. clone 整个仓库。
2. Docker build `nsr-agent`。
3. `docker run --net=host` 启动 Agent 容器。

#### 链路 B：`/api/install-agent.sh`

特点：

1. 这是 `scripts/install-agent.sh`。
2. 它不 clone 整仓库。
3. 它从 Center 下载 `agent-package.tar.gz`。
4. 在目标机创建 venv。
5. 安装 Python 依赖。
6. 写 `run.sh`。
7. 写 systemd service。

也就是说：

1. 根目录 `install.sh` 是“完整仓库 + Docker”路线。
2. `/api/install-agent.sh` 是“轻量打包 + venv + systemd”路线。
3. 两套安装策略同时存在。

### 14.4 热更新脚本

`update.sh` 的真实策略：

1. 读取旧版本。
2. `git pull`。
3. 热拷贝 `server/`、`scripts/`、`manage.py`、`version.py` 到容器。
4. 若检测到上一个提交里有 `web/` 变动，则 `docker compose up -d --build backend`。
5. 否则直接重启容器。

### 14.5 CLI

`manage.py` 当前支持：

1. `create-admin`
2. `remove-admin`
3. `reset-password`

## 18. 当前已知不一致与风险

这部分非常重要。下面列的是“现在真的存在”的问题，不是历史问题回放。

### 18.1 README 版本和内容滞后

现状：

1. README 仍显示 v0.133。
2. 代码版本号已经是 0.134.0。
3. README 的协议说明上半部分已写入 MTR，但下方“支持的探测协议”表仍只有 5 类协议，没有 MTR。

影响：

1. 新人先看 README 会得到一个半更新状态。

### 18.2 节点部署命令当前存在明显错误

现状：

1. `/api/install-agent.sh` 实际返回的是 `scripts/install-agent.sh`。
2. 这个脚本只接受 `--server --port --node-id --token ...` 这种参数。
3. 但 `server/api/nodes.py` 当前生成的脚本安装命令是：
   `bash <(curl -sL http://host:9191/api/install-agent.sh) agent --server ...`
4. 其中多出来的 `agent` 子命令只适用于根目录 `install.sh`，不适用于 `scripts/install-agent.sh`。

影响：

1. 节点管理页里“方式一：一键安装”这条命令现在会失败。
2. 这属于高优先级问题。

### 18.3 `scripts/install-agent.sh` 的默认端口与实际架构不一致

现状：

1. 该脚本默认 `PORT="9191"`。
2. 但当前 Agent 实际应该连 9192。
3. 节点管理页生成命令时会显式传 `--port 9192`，因此该路径可部分绕开问题。

影响：

1. 如果有人直接手工调用这个脚本且不传 `--port`，会走错端口。

### 18.4 `install.sh` 的就绪检测写法是错的

现状：

1. `install.sh` 用 `curl -sf http://localhost:9191/api/auth/login` 轮询服务就绪。
2. 但 `/api/auth/login` 当前只允许 `POST`。
3. 这会让它把 405 当成未就绪，直到超时。

影响：

1. 服务实际可能已经启动，但安装脚本仍会误报等待超时。

### 18.5 v0.131 所说的 range 校验，在当前代码里并不存在

现状：

1. `project-files/archive/update_v0.131.md` 写了 `range` 参数正则校验。
2. 但 `server/api/data.py` 当前没有这层校验。
3. `_parse_range_to_seconds()` 直接 `int(time_range[:-1])`。

影响：

1. 非法 range 仍可能触发 500，而不是预期的 400。

### 18.6 InfluxDB 定时下采样仍只聚合 3 个字段

现状：

1. `scripts/setup-influxdb.py` 只把 `latency`、`packet_loss`、`jitter` 写入聚合 task。
2. v0.131 更新日志声称已补齐 success、HTTP 各阶段耗时等字段。
3. 当前实际代码没有兑现这点。

影响：

1. 聚合桶中的 HTTP 阶段数据并不完整。
2. `success` 也不在定时 downsampling task 里。
3. 服务端虽然有 raw 回退聚合逻辑，但预计算聚合桶本身仍不完整。

### 18.7 系统设置页里大部分配置并不会驱动运行时

现状：

1. 页面允许改心跳窗口、离线阈值、快照推送间隔、默认探测参数等。
2. 但实际代码仍使用硬编码常量。
3. 当前只有 `site_title`、`site_subtitle` 被真正消费。

影响：

1. 设置页会给人一种“这些参数已经生效”的错觉。

### 18.8 MTR 详情页与 update_v0.133 描述不一致

现状：

1. 更新日志写的是“逐跳表 + 同时保留上方时序图表”。
2. 当前代码在 MTR 模式下直接跳过 ECharts，只显示终端表。

影响：

1. 如果有人按更新日志理解，会以为 MTR 还有趋势图，实际没有。

### 18.9 前端 API 类型给了 `start/end`，后端并未支持

现状：

1. `web/src/api/data.ts` 的 `getTaskData()` 参数里有 `start`、`end`。
2. `server/api/data.py` 实际只读取 `range`。

影响：

1. 这是一个半成品接口痕迹。
2. 当前没有真正的自定义时间区间查询。

### 18.10 `token_plain` 是当前设计权衡，不是最佳安全实践

现状：

1. 当前为了让部署命令可重复查看，数据库保留了明文 token。
2. 这是与早期“只显示一次 token”策略相反的当前实现。

影响：

1. 部署体验更好。
2. 但安全边界变弱。
3. 需要严格保证日志、接口、页面不额外泄露这类数据。
