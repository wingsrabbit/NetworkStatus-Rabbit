## 1. 项目概述

### 1.1 是什么

NetworkStatus-Rabbit 是一个独立的网络质量监控平台，用来观察不同节点之间，以及节点到外部目标之间的连通性和质量。

它主要回答三类问题：

- 某个节点到另一个节点的延迟、丢包和抖动如何
- 某个外部目标在最近一段时间是否稳定
- 哪些链路正在异常，哪些异常已经恢复

### 1.2 不做什么

- 不做服务器资源监控，例如 CPU、内存、磁盘
- 不做对外开放的公共 API
- 不做全网格自动互探测，避免节点规模扩大后任务失控

### 1.3 灵感来源

| 项目 | 借鉴方向 |
|---|---|
| Network-Monitoring-Tools | 底层探测协议实现 |
| ServerStatus-Rabbit | UI 风格、后台管理思路、Docker 部署模式 |
| SmartPing | 监控图表与时间范围交互方式 |

## 2. 核心概念

| 概念 | 含义 |
|---|---|
| Center | 管理服务端，负责后台、API、告警和数据汇总 |
| Agent | 部署在节点上的无头客户端，负责执行探测任务 |
| Probe Task | 一条具体探测任务，例如“节点 B 到节点 C 的 ICMP” |
| 内部目标 | 系统内的其他 Agent 节点 |
| 外部目标 | 系统外的 IP、域名或 URL |
| Protocol | 探测协议，初版为 ICMP、TCP、UDP、HTTP、DNS |
| Label | 节点标签，用于分组和筛选 |
| Downsampling | 对时序数据做分钟级、小时级聚合 |

## 3. 架构总览

### 3.1 整体架构

项目采用“集中式管理 + 分布式探测”的结构：

- Center 负责统一存储、统一告警、统一管理
- Agent 负责实际探测和结果上报
- 前端只连接 Center，不直接从各节点拉数据

### 3.2 集中式架构优势

- 所有数据集中存储，Dashboard 查询路径稳定
- 告警只在一个地方判断，逻辑简单且一致
- 节点和任务管理统一在一个后台完成
- 节点离线时仍可本地缓存数据，Center 恢复后再补传

### 3.3 探测节点本地存储概括

- 正常状态下，Agent 实时上报探测结果
- Center 不可达时，结果先写入 Agent 本地 SQLite
- 重连后按批次补传，Center 确认后再标记为已完成
- 已确认数据只保留 3 天，之后定时清理

## 4. 后端架构组成

| 模块 | 作用 |
|---|---|
| Flask | 提供内部 REST API |
| Socket.IO 服务 | 负责 Agent 接入、任务同步和前端实时推送 |
| InfluxDB | 存储探测时序数据 |
| SQLite | 存储节点、任务、用户、告警和系统设置 |
| 告警引擎 | 根据窗口规则判断是否告警或恢复 |
| Nginx | 统一承接 Web 页面、API 和 WebSocket 入口 |

后端的核心职责只有三件事：接节点、存数据、做判断。

---

## 5. Agent 架构组成

| 模块 | 作用 |
|---|---|
| WebSocket 客户端 | 与 Center 保持长连接 |
| 任务调度器 | 按间隔执行探测任务 |
| 探测引擎 | 统一封装 ICMP、TCP、UDP、HTTP、DNS 探测 |
| 本地缓存 | 保存未确认的数据，支撑断线补传 |
| 能力发现 | 启动时检测当前机器支持哪些协议 |

Agent 的目标不是复杂，而是稳定：能接任务、能探测、能补传。

## 6. 前端架构组成

前端使用 Vue 3 + TypeScript + Naive UI + ECharts，承担四类职责：

- 展示 Dashboard 总览
- 展示单任务的时序详情
- 提供节点、任务、告警、用户、系统设置后台页面
- 接收实时推送并做轻量交互反馈

界面重点不是炫技，而是让管理员能快速回答两个问题：哪里坏了，什么时候开始坏。

## 7. 数据与存储概览

### 7.1 时序数据

- 探测结果写入 InfluxDB
- 原始数据保留 3 天
- 分钟级聚合保留 7 天
- 小时级聚合保留 30 天

### 7.2 配置数据

SQLite 保存配置型信息，包括：

- 节点
- 探测任务
- 用户
- 告警通道
- 告警历史
- 系统设置

### 7.3 本地缓存

每个 Agent 维护独立 SQLite 缓存，用于保存未收到 ACK 的探测结果。

字段级结构、表定义和保留规则统一放在 [project-data.md](project-data.md)。

## 8. 通信与同步概览

### 8.1 节点接入流程

Agent 启动后执行以下流程：

1. 连接 Center 的 Socket.IO 服务
2. 发送节点身份、Token、配置版本和能力信息
3. Center 验证身份并比较配置版本
4. 如版本落后，Center 下发全量任务同步
5. Agent 应用任务后开始心跳和探测

### 8.2 数据上报与断线补传

- 每条结果都有唯一 result_id
- Center 入库成功后返回 ACK
- Agent 只有在收到 ACK 后才把结果标记为完成
- 断线期间积压的数据在重连后批量补传
- 超过 60 秒的历史数据只入库，不进入实时告警

### 8.3 节点在线判定

- Agent 每秒发送一次轻量心跳
- Center 维护最近 120 秒的心跳窗口
- 窗口内收到 20 个及以下心跳时，节点判定为离线

### 8.4 Dashboard 实时推送

- 节点状态变更即时推送
- 任务总览按秒推送聚合快照
- 详情页只在用户订阅对应任务时推送细粒度数据

完整事件名、JSON payload 和异常场景见 [project-data.md](project-data.md)。

## 9. 状态与权限

### 9.1 节点状态

节点只维护四种状态：

- registered：已注册但从未成功连接
- online：在线
- offline：离线
- disabled：被管理员禁用

### 9.2 告警状态

每条告警规则只维护三段状态：

- normal：正常
- alerting：告警中
- recovering：恢复中

### 9.3 用户角色

系统只保留两个角色：

- admin：可以管理节点、任务、用户、告警和系统设置
- readonly：只能查看数据，不能进行写操作

### 9.4 权限边界

项目把一些高风险行为直接写死为硬规则：

- admin 不能通过 Web 删除其他 admin
- admin 不能通过 Web 降级其他 admin
- admin 不能删除自己，也不能把自己降级
- 系统始终必须保留至少一个 admin

### 9.5 登录安全

- 登录成功后使用 httpOnly Cookie 保存 JWT
- Cookie 使用 SameSite=Strict
- Cookie Path 固定为 /
- WebSocket 与 REST 共用同一套 Cookie 认证

## 11. 探测能力

### 11.1 支持协议

初版支持五种协议：

- ICMP
- TCP
- UDP
- HTTP / HTTPS
- DNS

### 11.2 插件化设计

每种协议都是一个独立插件，统一接入调度器和结果结构。这样做的意义很直接：

- 新增协议时不需要重写调度框架
- 前后端只需要认识统一的结果格式
- 能力发现也能统一处理

### 11.3 能力发现

Agent 启动时会自检本机支持哪些协议，并把结果上报给 Center。自检失败不会阻塞 Agent 启动，只会把对应协议标记为不支持。

### 11.4 支持平台

- 正式支持 Linux 主机和 Docker 模式
- 正式覆盖 Debian / Ubuntu、CentOS / Rocky / Alma、Alpine
- Windows 原生不支持
- macOS 和 WSL2 不保证兼容

## 12. 告警机制

### 12.1 告警策略

告警不是单次触发，而是窗口化评估：

- 看最近 N 次结果，而不是只看当前一次
- 达到阈值次数才触发告警
- 连续正常达到恢复次数才发送恢复通知
- 冷却期内不重复轰炸

### 12.2 通知方式

告警结果会进入三条路径：

- 写入告警历史
- 通过 Webhook 发给外部系统
- 实时推送到前端

## 13. 部署概览

### 13.1 Center 部署

Center 推荐使用 Docker Compose，最小服务组为：

- nginx
- backend
- influxdb

### 13.2 Agent 部署

Agent 支持两种部署方式：

- Docker 运行
- 一键脚本安装到 Linux 主机

### 13.3 运行边界

- 对外只暴露一个端口 9191
- InfluxDB 只在内部网络通信，不直接暴露
- 节点 Token 允许在 MVP 阶段通过命令行参数传入，但不能写入日志，也不能在 UI 中重复展示
