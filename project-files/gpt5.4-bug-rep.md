# NetworkStatus-Rabbit v0.121 Bug Report

任务名称：debug NetworkStatus-Rabbit

报告版本：v0.121

报告时间：2026-03-24

结论级别：高置信度

说明：本报告合并了前两轮任务详情页问题与本轮新增需求/问题，作为 v0.121 统一问题报告。

---

## 1. 现状摘要

当前任务详情页已经能正常显示短周期数据，不再出现 `Invalid Date`，但仍然存在 8 类核心问题/需求：

1. 上方统计卡不会随实时数据持续刷新。
2. `30 分钟` 到 `24 小时` 的切换缺少可感知差异。
3. `7 天` 和 `30 天` 视图会混入秒级实时点，粒度错误。
4. `7 天` 和 `30 天` 的聚合数据链路缺少运行时保障，导致点数异常少或直接为空。
5. 前端数值展示没有统一两位小数规范，导致有时显示过长浮点数，有时又被错误截成 `0.0`。
6. 页面底部缺少全局状态栏，不能展示产品署名、GMT+8 当前时间和最近更新时间状态。
7. `7 天` 页面当前不会自动更新，需要切回其它范围或刷新页面后才会看到新数据。
8. `30 分钟` 到 `24 小时` 在历史不足时不应继续滚动成“看起来全都一样”的动态图，而应允许空白，明确告诉用户数据是从什么时候开始记录的。

你这次新增的问题里，除了两位小数规范外，还有两条需要直接写入产品行为规则：

1. 底部中间增加状态栏：`Powered by NetworkStatus-Rabbit · 现在时间（GMT+8）：YY/MM/DD · 最后更新：10秒内`。
2. 对 `30 分钟 ~ 24 小时` 的原始视图，如果该时间段前半部分还没有数据，就应该保留空白，不要把时间轴滚成“看起来永远只有刚开始这段数据”。

## 2. 问题一：统计卡不会随实时数据更新

### 2.1 现象

下方图表持续向前滚动时，上方的：

1. 平均延迟
2. P95 延迟
3. 平均丢包
4. 数据点数

仍然停留在进入页面或上一次切换时间范围时的值，不会跟着实时变化。

### 2.2 代码证据

文件：`web/src/views/TaskDetailView.vue`

首屏和切换范围时：

```ts
const [dataRes, statsRes] = await Promise.all([
  getTaskData(taskId, { range: range.value }),
  getTaskStats(taskId, { range: range.value }),
])
points.value = dataRes.data.data
stats.value = statsRes.data.stats
updateChart()
```

实时更新时：

```ts
function handleRealtimeUpdate(data: any) {
  if (data.task_id !== taskId) return
  const result = data.result as ProbeResult
  if (!result) return
  points.value.push(result)
  if (points.value.length > 500) points.value.shift()
  updateChart()
}
```

这里的行为是：

1. 只更新 `points.value`。
2. 只重绘图表。
3. 不重算 `stats.value`。
4. 也不重新请求 `/stats`。

### 2.3 结论

统计卡和图表走了两条不同的更新链路，所以现在的详情页在逻辑上是“半实时”的，不是完整实时。

## 3. 问题二：`30 分钟` 到 `24 小时` 切换缺少可感知差异

### 3.1 现象

切换 `30 分钟 / 1 小时 / 6 小时 / 24 小时` 时，页面经常看起来没什么变化。

### 3.2 代码证据

文件：`server/services/influx_service.py`

```python
if hours <= 24:
    return self.bucket_raw
elif hours <= 7 * 24:
    return self.bucket_1m
else:
    return self.bucket_1h
```

这意味着：

1. `30m`
2. `1h`
3. `6h`
4. `24h`

全部都走同一个 `raw` bucket。

文件：`web/src/views/TaskDetailView.vue`

```ts
watch(range, () => fetchData())
```

同时横轴统一格式化为：

```ts
dayjs(p.timestamp).format('HH:mm:ss')
```

### 3.3 结论

当前范围切换虽然会重新查询，但：

1. 数据源分辨率没变化。
2. 横轴表达没变化。
3. 页面没有显示当前窗口、采样粒度、实际起止时间。

所以用户非常容易判断成“切换没有生效”。

## 4. 问题三：`7 天 / 30 天` 视图混入了错误粒度的实时点

### 4.1 现象

你提供的截图已经能明确说明：

1. `7 天` 视图不应该继续按 3 到 4 秒一个点增长。
2. `30 天` 视图也不应该继续出现秒级点。

它们应该分别是：

1. `7 天` 按 1 分钟级别展示。
2. `30 天` 按 1 小时级别展示。

### 4.2 代码证据

后端确实想这么做。文件：`server/services/influx_service.py`

```python
if hours <= 24:
    return self.bucket_raw
elif hours <= 7 * 24:
    return self.bucket_1m
else:
    return self.bucket_1h
```

但前端在任何 range 下都执行相同的实时 append：

```ts
function handleRealtimeUpdate(data: any) {
  if (data.task_id !== taskId) return
  const result = data.result as ProbeResult
  if (!result) return
  points.value.push(result)
  if (points.value.length > 500) points.value.shift()
  updateChart()
}
```

### 4.3 结论

现在的逻辑是：

1. 先查长周期聚合数据。
2. 再无条件把秒级实时点塞进同一个 `points.value`。

这会直接破坏 `7d/30d` 的时间粒度语义，导致“历史聚合曲线 + 实时原始点”混流。

## 5. 问题四：`7 天 / 30 天` 数据点本身也不可靠

### 5.1 现象

从你这次截图看：

1. `7 天` 页面数据点数只有 7，明显偏少。
2. `30 天` 页面上方统计可能是 0，但下方图上仍然有线，说明“统计结果”和“图表点集”已经不一致。

这不是单纯的显示问题，而是长周期数据链路本身不稳。

### 5.2 代码证据

应用写入时只写 `raw`：

```python
self.write_api.write(bucket=self.bucket_raw, record=point)
```

仓库里虽然有 `scripts/setup-influxdb.py`，定义了：

1. `raw -> agg_1m`
2. `agg_1m -> agg_1h`

但应用运行时并不会：

1. 检查 downsampling task 是否存在。
2. 检查 `agg_1m` / `agg_1h` 是否持续产出。
3. 在长周期 bucket 不可用时做兜底或报错。

### 5.3 结论

因此 `7d/30d` 现在同时有两层问题：

1. 查询的数据源可能本身就不完整。
2. 即便历史数据正确，前端还会把秒级实时点继续混进去。

这就是你现在看到“7 天点数不对、30 天统计和图形对不上”的根因组合。

## 6. 问题五：前端小数位规范不统一

### 6.1 现象

你这次新增的问题非常明确：

1. tooltip 中出现了过长浮点数，例如：
   `0.21000000000000002`
   `0.034733333333333333`
2. 统计卡有时会显示 `0.0 ms`，这对像 `0.0421` 这样的值来说很奇怪。
3. 你要求的是：前端所有延迟类数据统一保留两位小数，例如 `0.0421 -> 0.04`，后端无需修改。

### 6.2 代码证据

文件：`web/src/views/TaskDetailView.vue`

图表 tooltip 当前没有 formatter：

```ts
tooltip: { trigger: 'axis' }
```

这意味着 ECharts 会直接展示原始浮点值，所以会把 JS 浮点误差直接暴露给用户。

同文件中，统计卡目前使用的是一位小数：

```ts
stats.avg_latency.toFixed(1)
stats.p95_latency.toFixed(1)
stats.avg_packet_loss.toFixed(1)
```

这会带来两个问题：

1. 精度标准不统一，与你要求的“两位小数”不一致。
2. 对于 `0.0421` 这样的值，`toFixed(1)` 会显示成 `0.0`，读起来像被吞掉了有效信息。

另外，文件：`web/src/views/DashboardView.vue`

```ts
return `${val.toFixed(1)} ms`
```

说明首页卡片也同样只保留一位小数，和你要求的统一两位小数规范不一致。

### 6.3 结论

当前前端至少存在三种不统一的数值展示行为：

1. 图表 tooltip 直接显示原始浮点数。
2. 统计卡显示一位小数。
3. 首页延迟卡也显示一位小数。

这会让页面同一份数据在不同区域看起来像三种不同结果。

## 7. v0.12 需要落实的前端数值规范

基于你这次的明确要求，v0.12 应补充一条统一规范：

1. 所有延迟类数值前端统一显示为两位小数。
2. 所有抖动类数值前端统一显示为两位小数。
3. 丢包百分比前端也建议统一为两位小数，避免和延迟展示风格割裂。
4. tooltip、统计卡、首页卡片、详情图表的标签和悬浮提示都必须使用同一套格式化函数。
5. 后端保留原始精度，格式化仅发生在前端显示层。

示例：

1. `0.0421 -> 0.04`
2. `0.2 -> 0.20`
3. `0.21000000000000002 -> 0.21`
4. `0.034733333333333333 -> 0.03`

## 8. 问题间的关系

当前这些问题是耦合的：

1. 统计卡不刷新，导致“实时图表”和“顶部摘要”分裂。
2. `7d/30d` 实时混流，导致长周期图表本身不可信。
3. 聚合桶缺少自检，导致长周期点数可能本身就不完整。
4. 前端格式化不统一，又把同一批数据展示成了不同精度和不同观感。
5. 关闭 `7d/30d` 的实时追加后，如果没有替代刷新机制，长周期页又会变成“必须手动切换或刷新才更新”。
6. `30m~24h` 如果继续只展示已有数据而不保留空白，就会让 `30m / 1h / 24h` 在历史不足时看起来几乎完全一样。

因此现在的问题不只是“显示不好看”，而是“同一页面不同区域、不同时间范围对同一份数据给出了不一致的解释”。

### 9.3 结论

这不是样式微调，而是一个新增的全局状态展示需求，应作为 v0.121 的新增功能项写入。

## 10. 问题七：`7 天` 页面现在不会自动更新

### 10.1 现象

你反馈“在 7 天内，我即使已经到 12 点 59 了，面板也不更新，需要切回来或者刷新”。

这个现象与当前代码是吻合的。

### 10.2 代码证据

文件：`web/src/views/TaskDetailView.vue`

现在已经有一层范围判断：

```ts
const isRawRange = computed(() => ['30m', '1h', '6h', '24h'].includes(range.value))
```

实时处理里直接写死：

```ts
if (!isRawRange.value) return
```

这意味着：

1. `30m / 1h / 6h / 24h` 允许实时追加。
2. `7d / 30d` 完全不做实时追加。

### 10.3 结论

之前的问题是：`7d/30d` 混入了秒级实时点。

现在的修正方向把这个问题截住了，但新的副作用是：

1. `7d/30d` 页面不再自动更新。
2. 用户必须切换范围或刷新页面，才能拿到新的聚合结果。

所以 v0.121 需要新增一条要求：

长周期视图虽然不能直接追加秒级原始点，但仍然需要一种自动刷新机制，例如按聚合粒度定时整页重拉。

## 11. 问题八：`30 分钟 ~ 24 小时` 在历史不足时应允许留空

### 11.1 新需求

你提出的规则是合理的：

对 `30 分钟` 到 `24 小时` 这些原始粒度视图，如果当前任务还没有积累满整个窗口的数据，就应保留前半段空白，让用户直接看到“数据是从什么时候开始记录的”。

### 11.2 这条规则要解决什么问题

你提到的核心疑问是：

1. 为什么我选择的是 `1 小时`，开始时间却也跟着增加？
2. 如果开始时间一直增加，岂不是会一直困在当前这段数据里？

这个判断本质上是在质疑当前图表把“滚动窗口”和“已有历史长度”混在一起展示。

在历史不足时，如果图表仍然只把现有点铺满整个画布，就会造成：

1. `30m / 1h / 24h` 看起来像同一张图。
2. 用户看不出到底是“窗口真的只有这么多”，还是“系统只记录了这么多”。
3. 时间轴的左边界持续前移，会让用户误以为自己永远被困在最新这一小段数据里。

### 11.3 结论

因此 v0.121 应明确写入一条行为规则：

1. `30m ~ 24h` 视图按所选窗口固定时间轴。
2. 如果窗口前半段没有数据，就保留空白。
3. 不要为了“铺满图表”而把已有数据拉伸成看似完整的整段历史。

如果这一条落实，你提到的“1小时开始时间也增加”的困惑基本会被消除，因为用户能直接看到窗口范围和数据起点不是一回事。

## 12. 修复建议

### P0：统一前端两位小数格式化

优先级最高的新要求是显示规范统一。

建议：

1. 抽一个前端通用格式化函数，统一处理 ms、百分比和空值。
2. `TaskDetailView` 的统计卡改为两位小数。
3. `TaskDetailView` 的 tooltip 增加 formatter。
4. `DashboardView` 的延迟卡改为两位小数。
5. 图表坐标轴 label 如需显示数值，也统一走两位小数。

### P0：实时更新时同步刷新统计卡

建议：

1. 每收到一条实时点，基于当前 `points.value` 重算统计。
2. 或者以节流方式重拉 `/stats`。

### P0：`7d/30d` 禁止直接混入秒级原始点

建议：

1. `30m~24h` 可以继续实时 append 原始点。
2. `7d/30d` 应关闭这种 append。
3. 但 `7d/30d` 不能因此完全静止，应按聚合粒度定时整页刷新。
4. 如果未来要支持 `7d/30d` 实时更新，应按聚合粒度补点，而不是直接塞原始点。

### P0：为全站布局增加底部状态栏

建议直接参考 `status.wingsrabbit.com` 的信息架构，在全局布局底部中间增加：

1. `Powered by NetworkStatus-Rabbit`
2. `现在时间（GMT+8）：YY/MM/DD HH:mm:ss` 或符合你要求的精简格式
3. `最后更新：10秒内 / N秒前`

并且：

1. `最后更新` 应基于当前页面最后一次收到数据或最后一次成功拉取数据的时间。
2. 小于 10 秒统一显示 `10秒内`。

### P0：`30m~24h` 在历史不足时固定窗口并保留空白

建议：

1. 选中 `30m / 1h / 6h / 24h` 后，时间轴应固定为对应窗口。
2. 如果窗口前半段没有数据，图表左侧应为空。
3. 不要只渲染“已有数据区间”然后把它铺满整个横轴。
4. 这样既能显示记录开始时间，也能避免 `30m~24h` 看起来总是一样。

### P1：增强长周期数据链路可观测性

建议：

1. 检查 `agg_1m`、`agg_1h` 是否存在。
2. 检查 downsampling task 是否运行。
3. 对 `7d/30d` 查询失败返回明确诊断，而不是静默空白或数据错乱。

### P1：增强范围切换反馈

建议：

1. 页面显示当前时间窗口。
2. 页面显示当前采样粒度。
3. 页面显示当前图表点数和实际时间范围。

## 13. v0.121 验收标准

1. 详情页打开后，图表和顶部四个统计卡都应随实时数据同步更新。
2. `30m / 1h / 6h / 24h` 切换后，页面应有明确可感知的窗口或粒度反馈。
3. `7d` 页面不再每 3 到 4 秒追加原始点，但仍会自动刷新，用户不需要手动切换或刷新页面。
4. `30d` 页面不再混入秒级实时点，并且统计与图形一致。
5. `30m ~ 24h` 在历史不足时，图表前半段应允许空白，能看出数据起始时间，而不是所有范围都像同一张图。
6. 底部中间新增状态栏，显示 `Powered by NetworkStatus-Rabbit · 现在时间（GMT+8） · 最后更新`。
7. `最后更新` 在小于 10 秒时统一显示为 `10秒内`。
8. tooltip 中不再出现长浮点串，例如 `0.21000000000000002`。
9. 所有延迟和抖动前端展示统一为两位小数，例如 `0.0421 -> 0.04`，不再出现 `0.0 ms` 这种误导性显示。

## 14. 最终结论

v0.121 的核心不只是“把图画出来”，而是把任务详情页的时间语义、刷新语义和状态语义统一起来。

当前最关键的问题是：

1. 统计卡、图表、长周期历史、实时数据流之间没有统一规则。
2. `7d/30d` 的数据粒度和点数都不可靠。
3. `7d/30d` 现在又因为禁用实时追加而失去了自动更新能力。
4. `30m~24h` 在历史不足时没有把“窗口范围”和“真实数据起点”区分开。
5. 页面缺少统一的底部状态栏，用户看不到当前时间和最近更新状态。
6. 前端格式化规则缺失，导致用户直接看到浮点误差或被错误四舍五入后的值。

如果不把这几项一起收口，任务详情页虽然可用，但仍然不具备稳定、可信、可解释的监控含义。
