## 🕒 第一部分：全量继承并确认的 16 项核心脱节遗留 (GPT-5.4 已指出，开发者烂尾未完)

### 1️⃣ 底层探测与驱动模块架构
**1. 欺骗性的 `agent/network_tools/` 挂载**
- **文档期望：** `PROJECT.md` - 11.4节，规定探测核心必须使用本地集成的 `Network-Monitoring-Tools` 外部模块。
- **实际脱节：** 仓库中现在的 `agent/network_tools/` 仅是以同名目录做了自身仓库自写的占位级降维代码（比如用 `subprocess` 跑系统的 ping），并非引入对齐上游能力的组件。

**2. HTTP 与 DNS 探测手段并未收敛**
- **文档期望：** HTTP 和 DNS 测试应该统一交由底层的 `curl_ping` 和 `dns_lookup` 去承载测量，与上游架构保持一致。
- **实际脱节：** `agent/probes/http_probe.py` 等代码随意直接调 `requests`/`curl`，根本没有往底层 `network_tools` 的标准链路里沉淀，违背架构分层。

**3. UDP 能力自检严重“降配”，无视系统依赖**
- **文档期望：** 能力发现中（11.5节），对于 UDP 会去用 `which nc` 指令严格判断当前宿主机是否部署了 `netcat` 探测环境以作为支持标准。
- **实际脱节：** `agent/probes/udp_probe.py` 当中仅仅粗暴地通过 Python 发起 `socket.AF_INET / SOCK_DGRAM` 创建。只要能获得普通本地端口描述符就自认为支持 UDP 能力，彻底抹杀了实际功能可不可行的强校验前置。

**4. 部署：半成品的一键部署安装脚本**
- **文档期望：** 一键部署必须提供立等可用的 Agent。
- **实际脱节：** `scripts/install-agent.sh` 的执行结果不能独立把节点带入在线态，在命令行终端的结束位置竟然硬编码打印让技术人员手动拷贝代码到目录。部署流断层。

### 2️⃣ 探针系统主状态机与缓存闭环
**5. 禁用节点的“斩草除根”语义被截断**
- **文档期望：** 系统对节点切换禁用态时需要切断并彻底清退停下一切轮询探测进程。
- **实际脱节：** 在 `server/api/nodes.py` 和 agent 的脱离机制中，只是一刀切断了 WebSocket 的连接。但是在被隔离的远端，由于未能调取 `stop_all()` 去注销探针，线程池里的任务依然向僵尸网络一样继续发着无效攻击。

**6. 任务同步状态机根本没有实质推进（pending 遗失）**
- **文档期望：** 中心在面临管理员修改/启停任务后，需将节点送入并等待 `pending` 握手后回归，或者超时去 `desync`，需要带版本的闭环保障防止下发丢失。
- **实际脱节：** `server/api/tasks.py` 增删改代码内仅仅单向让数据库里的 `config_version` 做了自增递进。而那些控制它进入同步重试保护环的 `mark_sync_pending()` 方法入口被遗落，发过去就结束了，状态形同虚设。

**7. 告警事件的状态机丢失 `recovering` 和存续时间**
- **文档期望：** 一次异常故障的开始和结尾会留档，而且存在恢复窗口态。并且明确 `alert_history` 表入库要求登记具体的恢复时长 `duration_seconds`。
- **实际脱节：** `server/services/alert_service.py` 开发者偷工捡料，只有 `normal` 和 `alerting` 两级倒转，而数据库写入层，对于持续时间这一溯源指标完全放了空。

**8. SQLite 本地断线重连“幽灵批次”问题（补传失控）**
- **文档期望：** Agent 因失联导致的数据重堆叠，在网络恢复后的“补发批次”（batch）中，必须登记好生命周期的 `batch_id` 和发送次数迭代。
- **实际脱节：** `agent/local_cache.py` 表里确实装模作样放了占位字段，但推送逻辑中根本没有任何一个地方给发出包裹写入归属批次码和增加 `retry_count`。只要出现大批量震荡补传没对齐，中心后端也无从判定丢的是什么周期的队列。

**9. InfluxDB 数据层“去重检查”降维缩水**
- **文档期望：** 为了抵御错乱发包重收的现象， `result_id` 要实现跨批次跨时空的强幂等持久校验。
- **实际脱节：** `server/services/influx_service.py` 被退化为由内存中的 `_DEDUP_TTL 600s` 来把门。服务端一旦被掐死重启，或者是稍微迟滞多了一点的发包，历史假重发就会让时序库中布满脏据记录。

### 3️⃣ C/S主次干线通信规范安全网
**10. 主链路 Websocket 私有化泛滥背离统一命名**
- **文档期望：** 上游明确制定的冒号分隔强协议字（例如 `agent:auth`, `agent:probe_result`）。
- **实际脱节：** 从 `agent/ws_client.py` 到后端的通信接收器，依旧全面违规地在使用私底下的底划线名（`agent_auth`）。

**11. Dashboard 的 WebSocket 管理脱轨与权限越界不作为**
- **文档期望：** 对于进入详情长连接，应该走标准的冒号事件，且当因过期/未鉴权的连接 `connect_error` 必须重定向登录处理。
- **实际脱节：** `web/src/composables/useSocket.ts` 代码随意，用了一层不一致的前缀名，同时对于被远端丢回的 `WS_AUTH_FAILED` 甚至只打了一个只在开发者调试界面出现的 `console.error`，没有做出页面拦截。

**12. Backend 读取权限门户大开**
- **文档期望：** 强制隔离 readonly 用户，拦截访问 `/admin/` 特权区的历史与设定。
- **实际脱节：** `server/api/alerts.py` 接口仅用普通的 login_required 放行。只读身份的马甲能够全副通关到管理员后台窥探修改告警频道配置，权限拦截没合拢。

### 4️⃣ 前端制图与展示缺失
**13. Frontend：缺失的兼容性能力动态预警告**
- **文档期望：** 前端 `TasksView.vue` 控制面板在新建任务敲击确认前，需立刻对选定的探测节点是否有这个探针能力做出对比反馈提示。
- **实际脱节：** 创建面板直接让一切操作都“蒙眼通过”，没有调用获取此能力对比的逻辑拦截。

**14. Frontend：管理列表缺乏直观的状态表现块**
- **文档期望：** `NodesView.vue` 里本该抓取获取到的节点环境字典 `capabilities` 来展示具体的灰/绿色能力检测标签。
- **实际脱节：** 该展示维度被阉割未制作，管理员无法在首页感知探针环境的健康与完整度。

**15. 前端制图：任务图表展现组件被极致同质化阉割**
- **文档期望：** `PROJECT.md` 10.3节与 10.4节有非常严肃的视觉说明：当节点发生报警，页面折线图底部必须要有红色阴影盖面高亮展示问题的跨越区间（即 ECharts 的 `markArea` 属性联动历史记录）。当前图层干净如洗，运维人员看告警犹如盲人摸象。
- **实际脱节：** 毫无绘制和挂钩告警历史相关的时序标注逻辑。

**16. 前端制图：专属协议图标严重降级**
- **文档期望：** HTTP需使用拆分的各阶段堆叠面积图（DNS+TCP+TLS+TTFB）表现长耗时根因，以及状态码散点群。
- **实际脱节：** `web/src/views/TaskDetailView.vue` 当中极尽敷衍。不管下面是 ICMP 还是 HTTP/DNS，统一硬塞普通的延迟/丢包折线，毫无对异构协议结构的定制化渲染分支。

## 🌟 第二部分：Gemini 3.1 Pro 独家追捕的隐蔽漏网项 (共 1 项)

**17. 系统全局副标题 / APP Title 与系统设定的断连级黑洞**
- **现象事实与对标：**
  系统内置了从接口配置系统界面标题信息的 `/admin/settings` 能力块。管理侧填好了这些定制化全局系统展现设定后，按系统交互常理及规格联动期望，此时页面标题应当改变，但实际上它没有跟网页和根元素做任何挂载。
- **实际脱节：**
  `web/src/views/LayoutView.vue` 模板以及路由导航（Router）拦截防线中，把 “NSR v0.x” 之流全部采用前端硬编码模式写死了。保存后的动态配置字段成了填进黑洞的数据。系统名称未按要求改变也无法触达 `document.title` 等框架级的底层呈现。
